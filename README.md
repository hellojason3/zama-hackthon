# Privyields

Confidential Qualified Yield Market built with Zama.

Privyields is a hackathon dApp for private allocation into yield products. A user proves they are qualified without revealing their full asset amount, wraps USDC into confidential cUSDC, allocates encrypted capital to a listed yield product, receives encrypted reward accounting, decrypts their own reward locally, and claims encrypted cUSDC.

## Core Flow

1. User generates and submits an Arkworks Groth16 asset-threshold proof: assets >= 1,000,000 USDC.
2. User mints mock USDC, approves the wrapper, and calls `ConfidentialUSDC.wrap`.
3. User encrypts the deposit amount with Zama Relayer SDK.
4. User calls `confidentialTransferAndCall` on cUSDC with the selected product id.
5. `ConfidentialYieldVault` receives the encrypted amount in its callback and records encrypted principal.
6. Product publisher updates APR in `YieldProductMarket`.
7. Vault accrues encrypted rewards using FHE arithmetic.
8. User requests reward decrypt access, decrypts through the Zama SDK, and claims encrypted cUSDC.

## Contracts

- `MockUSDC`: 6-decimal ERC-20 for demo minting.
- `ConfidentialUSDC`: OpenZeppelin confidential ERC-20 wrapper over mock USDC.
- `QualifiedInvestorGroth16Verifier`: generated BN254 Groth16 verifier exported from the Arkworks circuit.
- `Groth16QualifiedInvestorRegistry`: verifies asset-threshold proofs and marks qualified wallets.
- `MockQualifiedInvestorRegistry`: fallback demo registry used by legacy tests and mock-only flows.
- `YieldProductMarket`: 10-product marketplace and yield-rate publisher.
- `ConfidentialYieldVault`: confidential deposit receiver, encrypted principal/reward ledger, and encrypted cUSDC reward claims.

The installed OpenZeppelin confidential package exposes the current implementation as `ConfidentialFungibleToken` / `ConfidentialFungibleTokenERC20Wrapper`. The project design maps this to the ERC-7984 confidential asset rail concept: encrypted balances, encrypted transfers, and transfer-and-call callback deposits.

## ZK Scope

The ZK component is intentionally limited to qualification:

```text
private input: assetAmount, salt
public input: commitment, threshold, user, nullifierHash
statement: assetAmount >= threshold and commitment binds assetAmount to the user
```

For the live hackathon demo, the frontend calls a Next API route that runs the Rust Arkworks prover, receives a Groth16 proof JSON, lets the issuer/deployer register the commitment, and then submits the proof to `Groth16QualifiedInvestorRegistry`. The exact asset amount remains off-chain. This is still a demo attestation model: the commitment registration step stands in for a real bank, custodian, or compliance issuer.

The Arkworks setup is generated by a deterministic demo script for repeatable judging and testing. It is not a production trusted setup ceremony. The legacy `circuits/` directory contains the earlier Circom design sketch; the connected verifier path for this demo is the Rust Arkworks implementation under `zk/qualified-investor`.

## Frontend

The Next app is a single-screen operating console:

- wallet and qualification state
- 10-product yield market
- mint/approve/wrap cUSDC controls
- encrypted deposit via Zama Relayer SDK
- publisher APR update and encrypted reward accrual
- reward decrypt and encrypted claim

Set these environment variables after deployment:

```bash
NEXT_PUBLIC_MOCK_USDC=0x...
NEXT_PUBLIC_CUSDC=0x...
NEXT_PUBLIC_MARKET=0x...
NEXT_PUBLIC_VAULT=0x...
NEXT_PUBLIC_QUALIFICATION_REGISTRY=0x...
NEXT_PUBLIC_GROTH16_VERIFIER=0x...
NEXT_PUBLIC_REGISTRY_MODE=groth16
NEXT_PUBLIC_ZK_ASSET_THRESHOLD=1000000000000
```

## Commands

```bash
npm install
npm run compile
npm test
npm run typecheck
npm run build
npm run deploy:sepolia
npm run smoke:localhost
npm run server:local
./deploy-server.sh
DEPLOY_NETWORK=sepolia ./deploy-server.sh
./zk-qualified-demo.sh
npm run dev
```

For Sepolia deploys, configure a deployer key plus an RPC endpoint. A single private key is preferred for this demo wallet:

```bash
npx hardhat vars set DEPLOYER_PRIVATE_KEY
npx hardhat vars set INFURA_API_KEY
```

You can also use `MNEMONIC` instead of `DEPLOYER_PRIVATE_KEY`, or provide a full `SEPOLIA_RPC_URL` instead of `INFURA_API_KEY`.

## Demo Notes

- Local Hardhat tests cover product setup, vault-only deposit recording, the fallback demo qualification registry, and the Arkworks Groth16 registry verifier path.
- Local smoke mode: run `npm run chain`, `npm run deploy:localhost`, then `npm run smoke:localhost`. This validates deployment, product listing, qualification, mock USDC minting, approval, and rate publishing.
- Server-local mode: run `npm run server:local` on a VPS to start a local Hardhat chain, deploy contracts, write `.env.local`, and start the frontend. Prefer SSH tunnels for the frontend and RPC ports instead of exposing the Hardhat RPC publicly.
- One-command server update: run `./deploy-server.sh`. The script syncs the repo to `zama-hackthon:~/privyields`, prepares Arkworks verifier assets, installs dependencies, compiles contracts, restarts the local demo chain, deploys contracts, writes `.env.local`, builds the Next app, and starts `next start` on port 3000 for Caddy to proxy.
- Sepolia server update: run `DEPLOY_NETWORK=sepolia ./deploy-server.sh`. Contracts are deployed from the local machine using `DEPLOYER_PRIVATE_KEY` or `MNEMONIC`, then the server receives the deployment artifacts, frontend, prover binary, and proving keys. The frontend generates Groth16 proofs through `/api/zk/prove` and submits them to Sepolia with the connected wallet.
- Sepolia demo funding does not require real USDC. `MockUSDC.mint` is part of the deployed demo, so run `DEMO_WALLET=0x... npm run mint:demo:sepolia` after deployment to mint 1,000,000 mock USDC to your demo wallet.
- Arkworks ZK path: run `./zk-qualified-demo.sh` to generate BN254 Groth16 proving/verifying keys, produce an asset-threshold proof, verify it locally in Rust, export a Solidity verifier, compile it, and enable the Groth16 registry test.
- FHE transactions require the Zama Sepolia coprocessor contracts; use Sepolia for the live encrypted flow.
- The vault-custody model keeps funds in `ConfidentialYieldVault`. Products are strategy labels and rate publishers, not external custodians.
- Rewards are paid as encrypted cUSDC from the vault reserve for demo purposes.
